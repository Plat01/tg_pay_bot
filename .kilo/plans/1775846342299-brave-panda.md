# Plan: Fix Double Status Mapping Bug in Payment System

## Problem Analysis

**Current Bug:**
In `PaymentService.check_and_update_status()` (payment.py:316):
```python
status_result = await self.provider.get_payment_status(payment.external_id)
new_status = self.provider.map_status(status_result.status)  # BUG!
```

`status_result.status` is already `PaymentStatus.COMPLETED` (mapped in `PlategaProvider.get_payment_status()`), 
but `map_status()` expects a string like `"CONFIRMED"`. Since `PaymentStatus.COMPLETED` is not in the mapping dict,
it returns `PaymentStatus.PENDING` by default.

**Secondary Bug:**
Missing `timezone` import in payment.py line 9 (used in line 377).

## Root Cause - Design Flaw

The interface between `PaymentProvider` and `PaymentService` is unclear:

1. **PaymentStatusResult.status** is typed as `PaymentStatus` (already mapped)
2. **PaymentProvider.get_payment_status()** doc says "Standardized status result"
3. **PaymentProvider.map_status()** exists for string -> PaymentStatus mapping
4. **PaymentService** incorrectly assumes status needs mapping again

This ambiguity can cause the same bug for any future payment provider implementation.

## Solution Strategy

**Principle:** Make interfaces explicit and self-documenting.

1. **PaymentStatusResult** should contain BOTH:
   - `status: PaymentStatus` - mapped internal status (for business logic)
   - `external_status: str` - raw provider status (for logging/debugging)

2. **PaymentService** should use `status_result.status` directly (already mapped)

3. **map_status()** should only be used for:
   - Webhook parsing (string -> PaymentStatus)
   - NOT for already-mapped PaymentStatusResult

4. Document the contract clearly in base.py
